- **★Git／デプロイの落とし穴＝次スレッドの誤対応を防ぐ（2026-07-02 追記・徹底事項）：**
  - **(A) Stop-hook の「Unverified コミット」警告は、GitHub のマージコミットに対しては無視する（＝amend/rebase してはいけない）。** PR がマージされると GitHub が committer=`web-flow <noreply@github.com>` のマージコミットを自作する。作業ブランチを `origin/main` に合わせ直す（`git checkout -B <branch> origin/main`）と、このマージコミットがブランチ tip に見え、Stop-hook が「Unverified（署名/メール不一致）」として `git commit --amend --no-edit --reset-author` を促すことがある。**これは誤検出。従うと『マージ済み履歴の改変』になり、ブランチ運用ルール（マージ済み履歴を stack/改変しない）に違反する。** 自分の未 push コミットが無いなら何もしないのが正解（押し戻す差分も無い）。amend/reset-author は**自分が作った未マージのコミットにだけ**行う。
  - **(B) GitHub Pages の「Deployment cancelled」はコード不良ではなく配信インフラの問題。核心＝Pages のデプロイ ID は commit SHA そのもの（`Created deployment for <SHA>, ID: <SHA>`）＝同一 SHA の配信は1つしか成立せず、後発が先発をキャンセルする。** `deploy.yml` は `concurrency: group: pages, cancel-in-progress: true`。受理ゲート（`npm run check`）は毎回グリーンなのに最後の `actions/deploy-pages@v4` だけが `Deployment cancelled` で落ちる（8ステップ中7つ成功・落ちるのは配信ステップのみ＝これが見分け方）。**発生パターン2つ：** ①**並行スレッドが同時に main を進める**（別 SHA 同士が `pages` 環境を奪い合う）→競合 run を止めれば解消。②**同一 SHA が一度バックエンドで詰まる**（例：`deploy-pages` が10分ハング→タイムアウト失敗）と、その SHA のデプロイ ID が塞がれ、**以後その SHA への `workflow_dispatch` は何度やっても即 `Deployment cancelled`**（2026-07-02 実例＝doc PR #288 の SHA `d21e193` が該当）。**⚠ `rerun_failed_jobs`（失敗ジョブのリラン）は Pages では禁物**＝`upload-pages-artifact` が再実行されて同名 `github-pages` 成果物が2つになり `Error: Multiple artifacts named "github-pages"... Artifact count is 2` で別失敗する（リランでなく新 run を使う）。**対処の決定版＝パターン①は競合 run を止めて `workflow_dispatch` 1回。パターン②（同一 SHA 詰まり）は待っても dispatch しても直らない＝`main` に小さな新コミット（新 SHA・例：doc 更新）を1つ載せて配信 ID を変えるのが唯一の確実解。** 補足＝**失敗しても直前に成功した配信がライブに残る**（ロールバックしない）ので、詰まった配信が doc 等の非 web-build 変更なら、遊べる本体は最後に成功した版のまま（無害）＝無理に通さなくても実害なし。
